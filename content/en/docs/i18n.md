# Internationalization

goodoc ships multilingual. URLs are locale-prefixed (`/en/…`, `/zh/…`), English
is the default, and a language switcher sits in the header. Because everything is
statically generated, no server or redirects are involved.

## How content is organized

Content lives under `content/<locale>/`:

```text
content/
├── en/            # default locale
│   ├── docs/
│   └── blog/
└── zh/
    ├── docs/
    └── blog/
```

A document present only in the default locale **falls back** to it in other
locales, so you can translate incrementally — untranslated pages still render
(in English) under `/zh/…`.

## Links inside content

Write links **locale-relative**:

```markdown
See [Writing Content](/docs/humanize/writing-content/).
```

At render time the current locale prefix and the deployment base path are added,
so the link resolves to `/en/docs/humanize/writing-content/` (or `/zh/…`).
Assets with a file extension (e.g. `/logo.svg`) get the base path only — they are
never localized.

## Adding a language

Three small edits and you have a new locale:

1. **`lib/i18n.ts`** — add the code to `locales`, plus its `localeNames` and
   `localeHtmlLang` entries.
2. **`lib/dictionaries.ts`** — add a `Dictionary` block of UI strings (sidebar
   labels, "On this page", view-switch, etc.).
3. **`lib/site-config.ts`** — add a `LocaleContent` block under `locales` (hero,
   features, quickstart, nav, footer copy).

Then drop translations under `content/<new-locale>/`. Anything you skip falls
back to the default locale. TypeScript will flag steps 1–3 until each is
complete, so you can't forget a piece.

## What's where

| Concern | Location |
| --- | --- |
| Locale list / default / helpers | `lib/i18n.ts` |
| UI chrome strings | `lib/dictionaries.ts` |
| Project copy (hero, features, nav…) | `lib/site-config.ts` (`locales`) |
| Content | `content/<locale>/{docs,blog}/` |
